# Codex / Claude Runtime Parity Contract

The machine-readable source is [`runtime-parity.json`](runtime-parity.json). `yarn workflow:runtime-parity` rechecks the live local GSD installs, pinned per-host exact skill name/tree-digest manifests, pinned project-instruction bytes, inert projections parsed from the generated capability-registry JSON literal, the shared-config digest, the closed SHA-256 manifest for the project verifier entrypoint and all of its local transitive modules (including `state-check.mjs`), and a runtime-neutral good/bad proof fixture. It never `require()`s, imports, or launches the installed GSD core; digest drift therefore fails without giving drifted core bytes execution authority. Its final freshness snapshot repeats every core, every file and directory identity in each governed skill tree, instruction, config, matrix, project-verifier, registry-bearing core, and smoke-fixture input. File evidence requires a single link and binds digest plus `dev`/`ino`/link-count/mode/size/mtime/ctime identity, so an off-tree alias or A-to-B-to-A substitution does not become fresh merely because bytes and mtime were restored. Any live-check failure forces `readiness: not_ready` even when the matrix has no blocking row. The result exposes `matrixSha256`, `projectVerifierBundleSha256`, and `evidenceSnapshotSha256`; these values are review evidence, not an independent trust root, so the approved maintenance commit and reviewed matrix bytes must remain the source-release boundary.

The matrix also pins the complete expected output of the project wiring checker. After the separately approved activation transition, the only accepted finding set is empty: both `gsd-planner` and `gsd-plan-checker` must bind exactly the tracked `nutrition-planning-proof` skill. Every additional finding or changed finding payload makes the parity check fail with a stale-matrix finding; a generic blocking status cannot mask a new cause. From the reviewed maintenance commit and activated local config, the intended deterministic result is `status: pass` with `readiness: not_ready`: the role-binding row is now equivalent, while the remaining entrypoint, containment, and telemetry blocking/deferred rows still prevent readiness.

Matrix statuses have narrow meanings:

- `equivalent`: the stated bounded claim has matching evidence.
- `intentional_difference`: host behavior differs by design and carries an explicit rationale.
- `blocking`: local tooling may exist, but the project cannot rely on the behavior yet.
- `deferred`: evidence requires an approved pilot or later representative workflow samples.

Equal core hashes prove only installed-core identity. They do not make shared `.planning/config.json` an authoritative host signal, prove skill injection, or erase Codex/Claude dispatch differences. Artifact provenance comes from the active lease token; the shared `runtime` key is a configuration projection only.

The current matrix deliberately leaves artifact-writer wiring, lease entrypoint coverage, model/effort propagation, and representative telemetry unresolved. [`runtime-governance.md`](runtime-governance.md) defines the implemented lease/provenance/telemetry controls, [`gsd-reenable-pilot.md`](gsd-reenable-pilot.md) bounds the still-unapproved behavioral pilot, and [`gsd-reenable-readiness.md`](gsd-reenable-readiness.md) is the tracked maintainer decision ledger. No matrix result or readiness-report status authorizes `.planning` mutation in this checkout, GSD resume, GitHub writes, production, migration, runtime refresh, Tunnel, smoke, merge, or tag actions.

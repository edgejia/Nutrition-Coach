<interfaces>
From `server/orchestrator/index.ts`: `guardNoMutationLoggingClaim(reply, didLogMeal, didMutateMeal, context)` currently allows summary/history context when `SUMMARY_OR_HISTORY_LOGGING_REFERENCE_PATTERN` matches. That pattern is too broad because bare words such as `今天`, `目前`, `共`, and `攝取` can appear in false new-log claims. Narrow the allow condition to explicit aggregate/listing shapes, not topic words.

From `server/routes/chat.ts`: `handleStreamingReply()` creates `noMutationClaimGuard` only when `!hasSummaryContext`. This lets suspicious summary-context model text stream to the client before `guardNoMutationLoggingClaim()` can replace it at finalization. The streaming path must hold or guard suspicious no-mutation text until it knows the reply is allowed aggregate/listing summary text or must be replaced.

From `tests/integration/chat-streaming.test.ts`: use the existing `StreamingLLMProvider.queueChatStream()` and real `fetch(... Accept: text/event-stream ...)` patterns to prove streamed chunks, done payloads, persisted history, and route logs. This is the required true streaming coverage.

From project skills: use Node built-in `node:test`, real SQLite fixtures, `MockLLMProvider`/`StreamingLLMProvider` DI, and `yarn` commands only. Harness changes under `tests/harness/scenarios/*.ts` must be verified with `yarn verify:harness -- <scenario>` and artifacts treated as generated evidence.

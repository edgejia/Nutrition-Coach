<interfaces>
From `server/orchestrator/index.ts`: `guardNoMutationLoggingClaim(reply, didLogMeal, didMutateMeal)` currently rewrites no-mutation replies when `NO_MUTATION_LOGGING_CLAIM_PATTERN` matches. Refine the guard so it distinguishes false new meal logging claims from legitimate summary/history/query language. `get_daily_summary` sets `logMealSummary` without setting `didLogMeal` or `didMutateMeal`; that path must still allow replies such as `今天已記錄 2 餐，共 900 kcal`.

From `tests/unit/orchestrator.test.ts`: use the existing `createOrchestrator` fixtures and `MockLLMProvider` queue patterns. Add regression tests near the current no-mutation logging-claim tests so the expected distinction is local and easy to maintain.

From `tests/harness/scenarios/image-log-failure.ts`: `parseLiveChunkText(rawSSE)` already rejects malformed JSON and missing token fields, but it currently allows empty tokens and does not require at least one non-empty chunk before `done`. Tighten this helper and ensure sub-scenario evidence records chunk count/non-empty status.

From `tests/harness/scenarios/image-log.ts`: the current cleanup proof reads `SCENARIO_UPLOADS_DIR` in `finally` after `scenarioCtx.close()` and before removing the temp dirs, then only proves the manual scenario teardown removed the directory. Add a route-level assertion before teardown that the staged upload file created for the request has already been removed by `server/routes/chat.ts` cleanup.

From `tests/harness/scenarios/protein-trust.ts`: `parseReplyText(rawSSE)` catches malformed chunk JSON and returns an empty string. Change it to throw like `image-log` so malformed chunk frames fail the scenario instead of producing false empty reply text.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> `server/routes/meals.ts` | Untrusted PATCH JSON carries `expectedMealRevisionId` and single-item mutation shape. |
| route -> transaction service | Route preflight asks storage for current revision identity before mutation-shape decisions. |
| route -> summary/realtime side effects | Rejected stale writes must not cross into summary recompute or publisher fan-out. |
| server response -> client refresh helper | Client receives committed mutation metadata where `dailySummary` may be absent. |
| client store/API -> visible meal rows | Client refreshes visible rows as UX support while server remains write authority. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-05-01 | Tampering | `server/routes/meals.ts` PATCH preflight | mitigate | Call read-only expected revision assertion before grouped item-count rejection and before write/summary/publish paths. |
| T-62-05-02 | Tampering | `server/services/meal-transactions.ts` preflight | mitigate | Reuse existing device-scoped active transaction lookup and `MealRevisionPreconditionError`; do not duplicate route-only stale logic. |
| T-62-05-03 | Elevation of privilege | revision preflight | mitigate | Keep lookup constrained by signed guest-session `deviceId` and `mealId`; foreign meals still resolve as `MEAL_NOT_FOUND`/404 through existing service behavior. |
| T-62-05-04 | Information disclosure | conflict response and test proof | mitigate | Return/assert only stable `error`, `mealId`, `affectedDate`, and `currentMealRevisionId`; do not persist raw prompts, user text, assistant final text, tool payloads, provider bodies, image data, session material, or database snapshots. |
| T-62-05-05 | Repudiation / Tampering | summary and realtime side effects | mitigate | Integration test counts `summaryService.getDailySummary` and `publisher.publishDailySummary` calls and requires zero calls on stale preflight failure. |
| T-62-05-06 | Tampering | `client/src/meal-edit-refresh.ts` | mitigate | Client helper refreshes rows after committed success but does not decide write authority or synthesize revision safety. |
| T-62-05-07 | Denial of service | same-day row refresh | accept | One `/api/meals` refresh per successful same-day edit/delete is bounded and already used by existing stale recovery behavior. |
</threat_model>

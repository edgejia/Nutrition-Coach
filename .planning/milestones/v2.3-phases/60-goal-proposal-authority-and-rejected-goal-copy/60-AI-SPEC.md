## 1. System Classification

**System Type:** Conversational tool-use system

**Description:**
Nutrition Coach uses an OpenAI-backed conversational orchestrator with backend tool contracts for nutrition logging and goal updates. Phase 60 makes goal-change recommendations and confirmations backend-authoritative: the model may route intent to tools, but pending proposal state, mutation authorization, final rejected/cancel copy, and realtime publish/no-publish outcomes are controlled by deterministic TypeScript code.

**Critical Failure Modes:**
1. Short consent text such as `好` mutates daily targets without explicit current-turn numeric values or a valid backend-owned pending proposal.
2. Expired, missing, consumed, mismatched, or canceled proposals mutate targets or publish `goals_update`.
3. Failed validation or authority outcomes show model-authored success-style prose instead of deterministic backend Traditional Chinese copy.
4. Goal proposal or rejection proof logs raw prompts, user text, assistant text, tool payloads, provider bodies, images, session material, or database snapshots.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Consumer nutrition, wellness tracking, and behavior-change coaching

**User Population:** Personal beta users who use Traditional Chinese chat and meal photos to log food, monitor daily calories/macros, and occasionally ask the coach to change daily targets.

**Stakes Level:** High

**Output Consequence:** A successful goal update changes the persisted daily calorie, protein, carbohydrate, and fat targets that drive dashboard budgets, realtime `goals_update` events, future coaching context, and the user's day-to-day eating behavior. A failed or canceled goal update must leave those targets unchanged while giving the user clear Traditional Chinese guidance.

### What Domain Experts Evaluate Against

Dimension: Goal-change consent and target ownership
Good (domain expert would accept): Daily targets change only when the current user turn contains explicit numeric targets or when short consent such as `好` applies one active, non-expired, backend-persisted proposal whose exact calories/protein/carbs/fat were already shown to the user.
Bad (domain expert would flag): The app treats agreement with assistant prose, stale chat context, a consumed proposal, or generic encouragement as permission to change targets.
Stakes: Critical
Source: Nutrition Coach GOAL-01 through GOAL-04; mHealth behavior-change research treats prompts/cues and goal feedback as intervention mechanisms, so unintended target changes alter the intervention itself.

Dimension: Nutrition safety envelope and wellness boundary
Good (domain expert would accept): Proposed or applied targets stay within product-defined nutrition ranges, are presented as wellness tracking targets, and avoid disease-treatment or medical nutrition therapy claims unless the product has explicit clinical scope and review.
Bad (domain expert would flag): The coach proposes extreme restriction, unsafe macro targets, disease-specific treatment instructions, or ignores medical/allergy/dietary constraints when making a goal recommendation.
Stakes: High
Source: Nutrition LLM studies identify personalization, clinical validity, contraindications, and omitted guideline elements as core nutrition-safety concerns; FDA mobile software guidance distinguishes general wellness/calorie monitoring from higher-risk medical-device functions.

Dimension: Rejected-goal copy in Traditional Chinese
Good (domain expert would accept): Rejection, expiry, mismatch, validation, and cancel paths use deterministic, neutral Traditional Chinese copy that says the proposal was not applied, avoids success-style wording, and tells the user how to provide new numbers or request a new recommendation.
Bad (domain expert would flag): The response sounds like the goal was updated, blames or shames the user, exposes internal reason codes, mixes Simplified/Traditional Chinese inconsistently, or lets the model rewrite backend failure facts.
Stakes: High
Source: Dietitian-response evaluations separate polished language from clinically valid, context-sensitive counseling; Phase 60 requires exact-copy assertions for backend-owned rejection/cancel language.

Dimension: Realtime and downstream budget integrity
Good (domain expert would accept): `goals_update` publishes only after target persistence succeeds, proposal creation/rejection/cancel never publishes, and the client-visible budget always matches the committed backend target state.
Bad (domain expert would flag): A rejected proposal updates the dashboard, a successful persistence lacks a publish, or a publish/recompute failure leaves a stale proposal available for replay.
Stakes: Critical
Source: Nutrition Coach v2.3 data-integrity requirements; consumer nutrition apps rely on consistent daily budgets for self-monitoring and feedback.

Dimension: Metadata-only privacy evidence
Good (domain expert would accept): Evaluation and proof artifacts contain only allowlisted operational metadata such as turn ids, tool names, outcomes, field names, and redacted summaries; no raw prompt, user meal text, assistant final text, image data, tool payload, provider body, session material, or database snapshot is stored.
Bad (domain expert would flag): A failed goal-turn test or trace captures the user's diet text, body/health context, uploaded image content, full proposed targets tied to raw utterances, or LLM/provider payloads for convenience.
Stakes: Critical
Source: FTC health-app guidance treats diet-app and fitness-app data as sensitive consumer health information; Taiwan PDPA includes healthcare data and indirectly identifying personal data within protected personal data.

### Known Failure Modes in This Domain

- Polished nutrition copy can mask missing or unsafe nutrition substance: responses may read empathetic and clear while omitting key guideline elements, constraints, or escalation boundaries.
- Personalization drift can turn a wellness coach into generic or inappropriate advice: calorie/macro proposals may ignore the user's current target state, dietary pattern, language, or medical boundary.
- Ambiguous consent can silently alter the behavior-change plan: a short reply such as `好` can apply the wrong daily target if the app relies on conversational prose instead of backend proposal state.
- Failure copy can become behaviorally misleading: if a rejected or canceled update sounds successful, the user may eat against a target that the app did not actually persist.
- Evaluation evidence can become a privacy incident: raw chat, meal photos, provider bodies, or database snapshots are tempting for debugging but directly conflict with the product's metadata-only health-data contract.

### Regulatory / Compliance Context

- HIPAA is not assumed for this consumer beta unless Nutrition Coach becomes a covered entity, business associate, or handles protected health information for one. That does not remove health-data obligations.
- FTC Act and FTC Health Breach Notification Rule are directly relevant if the product offers or maintains a personal health record or draws identifiable health information from multiple sources; FTC guidance explicitly uses diet and fitness apps as examples.
- FDA mobile medical app oversight is risk- and function-based. General calorie/macronutrient monitoring for wellness is lower risk; disease diagnosis, treatment recommendations, or control of regulated devices would move the product into a different review posture.
- Taiwan PDPA is relevant to Traditional Chinese/Taiwan-oriented usage because identifiable or indirectly identifiable healthcare-related data is personal data. The product's metadata-only trace rule is the right default for routine proof and observability.

## 2. Framework Decision

**Selected Framework:** Existing custom TypeScript orchestrator + OpenAI SDK

**Version:** OpenAI SDK 4.104.0 as currently used by the repo

**Rationale:**
Phase 60 is a scoped authority and deterministic-copy correction inside an existing production flow, not a new agent architecture. The repo already has the right primitives: OpenAI calls behind `LLMProvider`, typed tool contracts, SQLite `turn_states`, deterministic mutation receipts, mock/harness providers, and metadata-only traces. Adding LangGraph or OpenAI Agents SDK would increase migration risk without solving the core requirement.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangGraph TypeScript | Useful for broad checkpointed state graphs, but overbuilt for a single-device pending proposal lifecycle that already fits the existing `turn_states` and tool-contract pattern. |
| OpenAI Agents SDK | Provides agent primitives and tracing, but adopting it would bypass established orchestrator seams and test harnesses for little benefit in this phase. |

**Vendor Lock-In Accepted:** Partial — the repo already uses OpenAI through `LLMProvider`; Phase 60 should preserve that DI boundary and avoid introducing additional framework lock-in.

### Entry Point Pattern
```ts
import type { LLMProvider, ChatMessage } from "../llm/types.js";
import { executeTool, getToolDefinitions } from "./tools.js";

export async function runConversationalToolTurn(args: {
  llmProvider: LLMProvider;
  messages: ChatMessage[];
  deviceId: string;
  toolDeps: Parameters<typeof executeTool>[2];
  currentUserMessage: string;
  previousAssistantMessage?: string;
  signal?: AbortSignal;
}) {
  const tools = getToolDefinitions();
  const response = await args.llmProvider.chat(args.messages, tools, {
    signal: args.signal,
  });

  if (!response.toolCalls?.length) {
    return { source: "model" as const, reply: response.content ?? "" };
  }

  for (const toolCall of response.toolCalls) {
    const outcome = await executeTool(toolCall, args.deviceId, args.toolDeps, {
      currentUserMessage: args.currentUserMessage,
      previousAssistantMessage: args.previousAssistantMessage,
    });

    if (toolCall.function.name === "update_goals" && outcome.success) {
      // Phase 60 rule: mutation facts and user-visible receipts are backend-owned.
      return { source: "renderer" as const, reply: outcome.result };
    }
  }

  return { source: "tool_round" as const, reply: "" };
}
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `LLMProvider` | Repo-owned adapter over `openai.chat.completions.create`; runtime uses `OpenAIProvider`, tests inject mocks. | Keep OpenAI isolated from services and preserve deterministic tests. |
| `ToolContract<Args, Result>` | Single contract object containing LLM JSON schema, Zod runtime schema, optional source guard, executor, and redacted log summary. | Add `propose_goals` and update `update_goals` without scattering validation across route/orchestrator code. |
| Handwritten OpenAI tool schema | The JSON schema sent in `tools[].function.parameters`. | Describe tool arguments to the model; keep in sync with Zod because OpenAI validates shape only probabilistically unless strict mode is compatible. |
| Zod `safeParse` validation | Runtime authority for tool args before execution. | Reject malformed, out-of-range, or wrong-mode goal updates before persistence or publish. |
| Backend renderer + `turn_states` | Deterministic user copy plus expiring, one-per-device state. | Store pending goal proposals for 30 minutes and render proposal/rejected/cancel copy outside the model. |

### Common Pitfalls
1. **Do not enable OpenAI `strict: true` on the current optional-field tool schemas without refactoring.** OpenAI structured function calling requires `additionalProperties: false` and all schema fields to be listed as required; the existing `update_goals` contract intentionally accepts partial field updates.
2. **Do not treat assistant prose as authority.** The current source guard can compare numeric fields against current user text or previous assistant text, but Phase 60 requires backend-owned pending proposal state; short consent like `好` must pass a backend proposal predicate.
3. **Do not let controlled goal failures continue to a model rewrite.** If validation, proposal authority, mismatch, expiry, or cancel occurs, return deterministic renderer copy with `finalReplySource: "renderer"` and no later LLM round.
4. **Be careful with streaming.** The OpenAI Node SDK returns async chunks for `stream: true`; the repo's `chatRound` buffers tool-call deltas until `finish_reason === "tool_calls"`. Once content tokens are emitted, treat the turn as final prose, not as a place to execute mutations.
5. **Keep logs metadata-only.** OpenAI API errors expose status, request id, error type, and code; log those fields plus redacted contract summaries, not raw prompts, user text, assistant text, tool payloads, provider bodies, images, session material, or database snapshots.

## 4. Implementation Guidance

**Model Configuration:**
- Runtime model remains `config.orchestratorModel`, defaulting to `gpt-5.4-mini`.
- Keep the Chat Completions endpoint for Phase 60 because `OpenAIProvider`, `LLMProvider`, tool-call mocks, and streaming tests already target it.
- Recommended provider parameters when touching `server/llm/openai.ts`: `temperature: 0.2`, `max_completion_tokens: 700`, and `reasoning_effort: "low"` for goal-proposal turns. These reduce rambling, cap cost, and leave deterministic copy to backend renderers.
- Do not rely on model settings for safety. Authorization must live in `ToolContract` execution and `turn_states`, not sampling behavior.

**Core Pattern:**
Use the model for intent routing and numeric proposal drafting; use backend contracts for proposal persistence, confirmation authorization, mutation, publishing, and final copy.

```ts
import { z } from "zod";
import type { DailyTargets } from "../services/device.js";
import type { ToolContract } from "./tool-contract.js";

const GOAL_PROPOSAL_KIND = "goal_proposal";
const GOAL_PROPOSAL_TTL_MS = 30 * 60 * 1000;

const goalTargetsSchema = z.object({
  calories: z.number().min(500).max(8000),
  protein: z.number().min(0).max(400),
  carbs: z.number().min(0).max(1000),
  fat: z.number().min(0).max(300),
}).strict();

type GoalProposalPayload = {
  proposalId: string;
  targets: DailyTargets;
  createdAt: string;
};

const proposeGoalsSchema = goalTargetsSchema;
type ProposeGoalsArgs = z.infer<typeof proposeGoalsSchema>;

const updateGoalsSchema = z.union([
  z.object({
    mode: z.literal("current_turn_values"),
    calories: z.number().min(500).max(8000).optional(),
    protein: z.number().min(0).max(400).optional(),
    carbs: z.number().min(0).max(1000).optional(),
    fat: z.number().min(0).max(300).optional(),
  }).strict().refine(
    (args) => ["calories", "protein", "carbs", "fat"].some((field) => args[field as keyof typeof args] !== undefined),
    { message: "at least one explicit goal field is required" },
  ),
  z.object({
    mode: z.literal("latest_proposal"),
  }).strict(),
]);

function hasGoalConsent(message: string): boolean {
  return /^(好|可以|幫我更新|就這樣|用這組)$/i.test(message.trim());
}

function hasGoalCancel(message: string): boolean {
  return /^(不要|取消|先不用|no)$/i.test(message.trim());
}

function renderGoalProposalCopy(targets: DailyTargets): string {
  return [
    "我可以先幫你改成這組每日目標：",
    `• 卡路里 ${targets.calories} kcal`,
    `• 蛋白質 ${targets.protein} g`,
    `• 碳水 ${targets.carbs} g`,
    `• 脂肪 ${targets.fat} g`,
    "如果要套用，請回覆「好」；如果要調整，請直接給新的數字。",
  ].join("\n");
}

export const proposeGoalsContract: ToolContract<ProposeGoalsArgs, GoalProposalPayload> = {
  name: "propose_goals",
  description: "建立一組待使用者確認的每日目標提案；不更新每日目標。",
  parameters: {
    type: "object",
    additionalProperties: false,
    properties: {
      calories: { type: "number", minimum: 500, maximum: 8000 },
      protein: { type: "number", minimum: 0, maximum: 400 },
      carbs: { type: "number", minimum: 0, maximum: 1000 },
      fat: { type: "number", minimum: 0, maximum: 300 },
    },
    required: ["calories", "protein", "carbs", "fat"],
  },
  zodSchema: proposeGoalsSchema,
  logSummary: () => ({ tool: "propose_goals", proposedFields: ["calories", "protein", "carbs", "fat"] }),
  execute: async (targets, context) => {
    const deps = context.deps?.toolDeps as {
      turnStateService: {
        putState<T>(deviceId: string, kind: string, payload: T, ttlMs: number): Promise<void>;
      };
    };
    const deviceId = context.deps?.deviceId as string;
    const proposal: GoalProposalPayload = {
      proposalId: crypto.randomUUID(),
      targets,
      createdAt: new Date().toISOString(),
    };

    await deps.turnStateService.putState(deviceId, GOAL_PROPOSAL_KIND, proposal, GOAL_PROPOSAL_TTL_MS);

    return {
      ok: true,
      result: proposal,
      // This string should be produced by a backend renderer in implementation.
      toolMessage: renderGoalProposalCopy(targets),
    };
  },
};
```

**Tool Use:**
- Add `propose_goals` to `toolRegistry`. It persists the latest pending proposal and returns backend-rendered Traditional Chinese proposal copy. It must not call `deviceService.updateGoals` and must not publish `goals_update`.
- Change `update_goals` from "partial numeric args only" to explicit modes: `current_turn_values` and `latest_proposal` (or `proposal_id` only if planning proves hidden handoff is reliable). Empty args must remain invalid.
- For `current_turn_values`, keep numeric `sourceFields` protection so model-supplied numbers must appear in the current user turn or immediately previous clarification.
- For `latest_proposal`, ignore model-supplied target numbers and load the active backend proposal. Require current-turn consent and reject cancel/reverse terms before persistence.
- Add cancel handling close to the goal contract/orchestrator boundary: clear the active proposal, do not mutate, do not publish, and return neutral backend copy.

**State Management:**
- Store pending proposals in `turn_states` with kind `goal_proposal`, one active row per device, and `ttlMs = 30 * 60 * 1000`.
- A new `propose_goals` overwrites the previous pending proposal for the same device.
- `getState` expiry behavior already clears expired rows; treat `undefined` as unavailable and fail closed.
- Clear pending proposal state after successful target persistence, after explicit cancel, and when expiry is observed. Do not clear it for ordinary validation/source-guard failures that should remain retryable.
- Keep mutation ordering strict: validate/authorize first, persist with `deviceService.updateGoals`, then clear proposal, then publish `goals_update`. If publish/recompute fails after persistence, do not leave the proposal reusable.

**Context Window Strategy:**
- Do not depend on conversation history to carry authority. `loadHistory(..., 10)` is enough for conversational continuity, but proposal authority must come from `turn_states`.
- Update `system-prompt.ts` so ambiguous goal-change intent calls `propose_goals`; confirmations call `update_goals` in proposal mode; rejections/cancel terms do not call mutation tools.
- Keep proposal IDs out of user-facing copy. If a hidden id is used, pass it only through tool results/system-visible state and test that the next-turn model can use it reliably. Otherwise prefer the single-active latest-proposal mode.
- Keep deterministic rejected/cancel copy short enough that no model summarization is needed. This avoids context-window and rewrite drift.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with Pydantic

This TypeScript repo should use the Zod equivalent of a Pydantic model. The framework integration is already `ToolContract.parameters` for the OpenAI-facing JSON schema plus `ToolContract.zodSchema.safeParse()` for runtime validation. Do not add Python/Pydantic.

```ts
import { z } from "zod";

export const goalProposalOutputSchema = z.object({
  status: z.literal("proposal_created"),
  proposalId: z.string().uuid(),
  targets: z.object({
    calories: z.number().min(500).max(8000),
    protein: z.number().min(0).max(400),
    carbs: z.number().min(0).max(1000),
    fat: z.number().min(0).max(300),
  }).strict(),
  copySource: z.literal("renderer"),
}).strict();

export type GoalProposalOutput = z.infer<typeof goalProposalOutputSchema>;

export function parseGoalProposalOutput(raw: unknown): GoalProposalOutput {
  const parsed = goalProposalOutputSchema.safeParse(raw);
  if (!parsed.success) {
    throw new Error(`invalid goal proposal output: ${parsed.error.issues.map((issue) => issue.path.join(".")).join(",")}`);
  }
  return parsed.data;
}
```

- **OpenAI integration:** keep tool argument schemas in `tools[].function.parameters`; use `response_format: { type: "json_schema", ... }` only for future non-tool structured assistant JSON. Phase 60 should prefer tool contracts because mutations need backend services.
- **Validation retry policy:** zero LLM retries for mutation authority failures, validation range failures, expired/missing proposal, and cancel. Return deterministic renderer copy immediately. For non-mutating `propose_goals` schema mistakes, allow at most one additional tool round within the existing `MAX_ROUNDS = 3`, logging only `tool`, `failureReason`, and field names.
- **Surface condition:** if validation still fails after the bounded retry, surface generic backend fail-closed copy; never ask the model to explain internal schema errors.

### Async-First Design

- OpenAI calls, tool execution, SQLite-backed services, summary recompute, and realtime publish are all awaited through the orchestrator.
- The common TypeScript mistake is `array.forEach(async ...)` for tool calls. Keep the existing `for...of` sequence because mutation order, proposal clearing, and publish/no-publish invariants matter.
- Stream only final user-facing prose. Await tool calls and Zod validation before any mutation. For Phase 60 rejected/cancel/success goal copy, prefer immediate renderer replies over streamed model text.
- Keep `AbortSignal` plumbing through `LLMProvider` calls; aborted OpenAI requests should remain provider metadata, not user-visible implementation detail.

### Prompt Engineering Discipline

- System prompt owns routing instructions: when to call `propose_goals`, when to call `update_goals`, and that deterministic goal proposal/rejected/cancel copy must be preserved.
- User text is never trusted as structured data until the backend contract validates it. Current-turn numeric values are checked by source guard; proposal confirmations are checked against backend state and consent vocabulary.
- Use 2-4 inline Traditional Chinese examples in `goalUpdates`, not dynamic retrieval, because this is a small fixed policy surface.
- Set output budgets explicitly when provider config is touched: `max_completion_tokens` for Chat Completions and concise prompt wording. Production goal-copy paths should not require model-authored long-form output.

### Context Window Management

- This is conversational tool use, not RAG. Do not add vector search, embeddings, or reranking for Phase 60.
- Keep `loadHistory(chatService, deviceId, 10)` for conversational continuity and `getHistory(..., 3)` for local previous-assistant checks.
- Pending proposal facts live in `turn_states`, not in the prompt transcript. A short consent must confirm the backend row, not whatever the model remembers.
- If future history grows beyond budget, summarize only stable nutrition context and prior non-sensitive outcomes; never summarize raw prompts, exact user text, tool payloads, images, session material, or database snapshots into logs or traces.

### Cost and Latency Budget

- Before changing production cost assumptions, verify the active model and pricing against official OpenAI docs; pricing is intentionally not encoded as a locked value in this phase contract.
- Budget work should estimate three scenarios from current traces: proposal-only turn, proposal-confirmation turn, and rejected/cancel turn. Record token counts and model config in metadata-only summaries, not raw prompts or responses.
- Phase 60 should reduce cost by short-circuiting deterministic proposal, rejection, cancel, and success receipts through backend renderers instead of a second final-prose LLM round.
- Use exact-match deterministic handling for cancel/consent vocabulary and backend state checks. Do not semantic-cache mutation decisions because they are device-specific and stateful.
- If later routing/classification is split out, use a cheaper model only for non-authoritative intent hints. All mutation authorization must still be enforced by TypeScript contracts.

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Explicit `update_goals` mode and Zod range validation | Every `update_goals` tool call | Block mutation, do not publish, return deterministic validation copy for invalid mode, empty args, unknown fields, or out-of-range targets. |
| Backend proposal authority predicate | `update_goals` proposal mode or short consent terms such as `好`, `可以`, `幫我更新`, `就這樣`, `用這組` | Block mutation unless current-turn consent is present and a non-expired active backend proposal exists; return generic fail-closed copy on missing/expired/consumed/mismatched state. |
| Cancel/reverse-term handling | Current user message matches `不要`, `取消`, `先不用`, or `no` while a proposal may exist | Clear active proposal, do not mutate, do not publish, return neutral backend cancel copy. |
| Mutation-before-publish ordering | Any successful goal persistence path | Persist targets first, clear proposal after commit, then publish `goals_update`; never publish for proposal creation, validation failure, authority failure, or cancel. |
| Renderer-owned final reply enforcement | Any proposal, goal success, rejected-goal, validation, or cancel outcome | Short-circuit later LLM rewrite and record `finalReplySource: "renderer"` with plain deterministic copy. |
| Metadata-only trace/log redaction | Tool call receipt, trace event, harness artifact, route fallback, or proof artifact creation | Store only allowlisted operational metadata; drop raw prompts, user text, assistant final text, tool payloads, provider bodies, images, session material, database snapshots, device ids, and non-redacted target values. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| Rejected-goal exact-copy pass rate | Run full Phase 60 unit/integration suite on every PR and release gate. | Block merge until exact copy, forbidden-term, and renderer-source assertions pass. |
| Unauthorized mutation rate | Run behavior matrix and targeted integration cases; sample production traces where `goal_update_rejected`, `goal_update_success`, or `goals_update_published` appears. | Add failing fixture covering the trace pattern, then fix contract/orchestrator logic before release. |

| Proposal lifecycle drift | Batch-check sampled traces for proposal created/applied/canceled/expired sequences using metadata only. | Add lifecycle unit/integration coverage for the missing transition and review `turn_states` clearing order. |
| Privacy artifact cleanliness | Run redaction tests and inspect generated harness artifacts after `yarn verify:harness -- behavior-matrix`. | Delete contaminated artifacts, fix sanitizer/log summary, and rerun proof before sharing results. |
| Human copy/safety review findings | Review a small sample of deterministic copy and wellness-boundary examples before beta release and after material copy changes. | Update renderer strings or range/escalation guidance, then refresh exact-copy tests. |

## 7. Production Monitoring

**Tracing Tool:** Existing metadata-only `LlmTraceRecorder` and orchestrator hooks are the production baseline. If external visualization is needed, export the same redacted events to Arize Phoenix through OpenTelemetry; Phoenix must not receive raw chat, tool payloads, provider bodies, images, session material, database snapshots, or device ids.

**Key Metrics to Track:**
- `goal_proposal_created` count, with proposed field names only.
- `goal_update_success` count by mode (`current_turn_values` vs `latest_proposal`) and updated field names only.
- `goal_update_rejected` count by coarse failure reason (`validation`, `guard`, `proposal_unavailable`, `proposal_expired`, `proposal_mismatch`, `canceled`, `execute`) without raw text.
- `goals_update_published` count and its correlation to successful persistence only.
- Final reply source distribution for goal turns: renderer vs model vs fallback.
- Trace redaction violation count from automated denylist checks.
- Latency for proposal creation, proposal confirmation, validation rejection, and cancel paths.

**Alert Thresholds:**
- Page/release-block immediately if any rejected, canceled, expired, missing, or consumed proposal path mutates targets or publishes `goals_update`.
- Page/release-block immediately if any trace/proof artifact contains raw prompt, user text, assistant final text, provider body, tool payload, image data, session material, database snapshot, device id, or non-redacted target payload.
- Alert if `finalReplySource !== "renderer"` for any goal success, proposal creation, rejected-goal, validation failure, or cancel path.
- Alert if validation/authority rejection rate exceeds 10% of goal-update attempts over a rolling day; investigate prompt routing, copy clarity, and proposal lifecycle bugs.
- Alert if p95 latency for deterministic proposal/rejection/cancel paths exceeds 750 ms excluding model time, because those paths should be backend-owned and fast.
- Alert if `goals_update_published` count differs from successful persisted goal mutations by more than 1 event over a rolling hour.

**Smart Sampling Strategy:**
Sample 100% of goal traces with rejection, cancel, validation failure, fallback final reply, non-renderer final reply, multiple proposal transitions in one device session, or any redaction-denylist hit. Sample 10% of successful goal updates and 5% of proposal-only turns for trend review. Human reviewers see only metadata summaries plus approved deterministic copy identifiers unless the user explicitly flags a conversation under a future raw-debug capture design.

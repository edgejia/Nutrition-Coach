<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend computation -> turn_states | Only backend-computed numeric proposal payloads may be persisted for later approval. |
| turn_states -> approval flow | Stored proposals are short-lived, device scoped, and revision scoped. |
| renderer copy -> user | User-facing copy must not leak internal IDs, tool names, trace details, or policy-heavy jargon. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-66-05 | Tampering | `meal_numeric_correction_proposal` state | mitigate | Store proposal payloads only through backend service inputs and include expected meal revision for stale-write protection. |
| T-66-06 | Elevation of Privilege | cross-kind proposal coexistence | mitigate | Use distinct turn-state kinds so meal proposal creation does not clear goal proposals and same-kind replacement stays scoped. |
| T-66-07 | Repudiation | renderer proposal copy | mitigate | Copy shows concrete before/after fields so approval target is unambiguous without formulas by default. |
| T-66-08 | Information Disclosure | proposal/cancel/failure copy | mitigate | Unit tests use forbidden-term denylist and do not expose proposal IDs, internal fields, raw prompts, tool args, or trace payloads. |
| T-66-SC | Tampering | npm/pip/cargo installs | accept | No package installation is planned; Package Legitimacy Audit lists none. |
</threat_model>

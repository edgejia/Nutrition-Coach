| Property | Value |

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, compact metric gaps, row meta separation |
| sm | 8px | Compact control gaps, focus offset, grouped item gaps |

| xl | 32px | Major panel separation if needed |

Exceptions: Existing row-specific dimensions remain allowed: Home meal thumbnails stay 40px, History meal thumbnails stay 32px, icon touch targets stay at least 44px where already established, Meal Edit back button remains 38px inside a 48px header column, and existing 10px/12px/14px CSS gaps may remain where already present. Do not introduce new spacing values unless required to preserve the current layout.

| Element | Copy |
|---------|------|
| Primary CTA | `儲存餐點` |

- Ineligible Home rows: no new copy, no disabled label, no cannot-edit message.
- Grouped Home row destination: existing grouped-lock copy in `MealEditScreen`; do not add Phase 74 grouped edit copy.

## Phase Interaction Contract

| Surface | Contract |
|---------|----------|
| Home eligible meal row | Entire row is the activation target. Prefer native `<button type="button">` semantics matching History rows. Preserve the current row layout: thumbnail, meta, title, macros, calories. |
| Home ineligible meal row | Render the same visual row as today, without button semantics, click handler, keyboard handler, disabled styling, tooltip, chevron, or explanatory copy. |
| Keyboard | Eligible Home rows must be reachable by Tab and activate with native button Enter/Space behavior. If native button markup is impossible, use the existing `MessageBubble` pattern: `role="button"`, `tabIndex={0}`, and Enter/Space handling. |
| Pointer | Row hover/active treatment may match `sp-history-meal-row:hover`: stronger border and `var(--sp-surface-2)` background. Do not add a separate edit icon or chevron. |
| Focus | Eligible Home rows use `outline: 2px solid var(--sp-lime); outline-offset: 2px;`. Focus state must not resize the row. |
| Routing | Home rows call `openMealEdit(payload, "home")` with a complete authoritative `MealEditPayload`. Do not introduce a Home-specific edit screen, route, or store boundary. |
| Payload authority | Reuse `buildHistoryMealEditPayload(meal, todayDateKey)` or a safe can-build wrapper around it. Missing `mealId`, `mealRevisionId`, nutrition fields, `itemCount`, or `loggedAt` means no activation. |
| Grouped meals | Complete grouped Home rows still open Meal Edit. `MealEditScreen` owns the existing `itemCount > 1` grouped-lock branch. No direct grouped save/edit/delete controls are introduced in Phase 74. |
| Post-edit return | `onBack` returns naturally to Home. The header back button must expose `aria-label="返回首頁"` for Home-origin edits. |
| Mutation states | Save, delete, stale conflict, validation error, refresh, and grouped-lock states stay in `MealEditScreen`; Home does not add post-edit highlight or success cues. |
| Capability metadata | `client/src/contracts/capability-matrix.ts` is the source of truth. Home must reflect implemented `openMealEdit`; Day Detail must remain read-only and stop over-claiming Meal Edit entry. |

## State Contract

| State | Required UI |
|-------|-------------|
| Home has no meals | Keep existing `SportCard` empty state and `去對話記錄` action. |
| Home complete single-item meal | Row is visually consistent with current Home row and opens editable Meal Edit form. |
| Home complete grouped meal | Row is visually consistent with current Home row and opens existing grouped-lock Meal Edit state with `到對話修正`. |
| Home incomplete/fallback meal | Row is visually unchanged and read-only. Render path must not throw. |
| Meal Edit stale save/delete conflict | Keep existing alert copy, red destructive styling, and `重新載入餐點` recovery button. |
| Meal Edit generic failure | Keep existing non-success error copy; never imply mutation success when server write failed. |

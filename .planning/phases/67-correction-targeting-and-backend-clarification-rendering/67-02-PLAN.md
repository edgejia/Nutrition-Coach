<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM query text -> meal resolver | Untrusted model/user target text influences candidate scoping and ranking. |
| resolver -> turn state | Pending option state persists across turns and must remain device/action scoped. |
| resolver -> mutator tools | Resolved output authorizes later mutation only by meal id plus revision id. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-67-01 | Elevation of privilege | `loadActiveCandidates` and `findMeals` | mitigate | Keep every candidate query scoped by `deviceId`; never accept model-provided meal ids as resolver authority. |
| T-67-02 | Tampering | evidence-tier resolver | mitigate | Encode locked evidence tiers so recency or weak prompt text cannot retarget away from stronger label/period evidence. |
| T-67-03 | Tampering | pending selection state | mitigate | Store exact rendered options, action, and scope metadata; reject action mismatch and contradictory explicit evidence. |
| T-67-04 | Information disclosure | service clarification prompt | mitigate | Do not render raw correction request as meal label; omit calories/macros and inferred periods from option text. |
| T-67-05 | Tampering | candidate loading SQL | mitigate | Use existing Drizzle query builders; no raw SQL interpolation of query text. |
| T-67-SC | Tampering | package installs | mitigate | No dependency additions. |
</threat_model>

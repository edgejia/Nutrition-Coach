## Interaction Contract

### Same-Day Summary And Meal Rows

- For valid `daily_summary` events where `affectedDate` is today and `source` is `meal_mutation`, refetch today's meal rows before committing the incoming summary.
- If the row refetch succeeds and the coordinator token is still latest, commit rows and summary together so the Home/Summary totals and visible meal rows describe the same fresh state.
- If the row refetch fails, commit nothing from that SSE event. Keep the previously visible summary, rows, loading state, and navigation state unchanged.
- If initial `getMeals()` and SSE reconcile overlap, the latest coordinator token wins. Older row-load results must not overwrite newer rows or a newer summary/row pair.
- First `source: "initial"` events with no loaded rows may preserve the existing guarded initial summary behavior. Reconnect `source: "initial"` events with loaded rows must use the same refetch-first/drop-on-failure semantics as same-day mutation events.

### Historical Affected-Date Invalidation

- For valid non-future `affectedDate` values that are not today, never call `setDailySummary`, never refresh today's rows, and never enter same-day reconcile.
- Record historical invalidation through the existing `recordMealMutation(affectedDate)` / `lastMealMutation` nonce.
- Refresh only matching visible historical surfaces: an open Day Detail whose `dateKey` equals `affectedDate`, or the active History screen when `affectedDate` is the selected day or falls inside the visible history week.
- Do not refresh just because the History tab exists. Non-visible historical dates receive passive invalidation only.
- Historical refresh results must be latest-event-wins per affected date. Older in-flight historical refreshes must not overwrite newer refresh results for the same visible date.

### Invalid And Out-Of-Scope Events

- Invalid JSON, invalid envelope shape, invalid source, invalid `DailySummary`, impossible calendar dates, `summary.date !== affectedDate`, and product-out-of-scope future dates mutate no state.
- Invalid events must not trigger `setDailySummary`, row refresh, same-day reconcile, `recordMealMutation`, Day Detail refresh, History refresh, or `lastMealMutation` nonce increments.
- Invalid and ignored events stay silent in the UI and follow the existing `goals_update` malformed-frame precedent.

## Visual Contract

- Preserve the existing operational, dense, mobile-first sport UI: `var(--sp-bg)` canvas, `var(--sp-surface)` cards/rows, `var(--sp-line)` borders, and compact history/meal layouts.
- Do not introduce landing-page composition, decorative cards, nested cards, gradient orbs, broad visual polish, or new visual hierarchy.
- Do not add visible freshness indicators. No stale badges, warning chips, toast notifications, banners, spinners specifically for SSE reconcile, inline timestamps, or row-level refresh markers.
- Historical visible refresh may reuse existing loading affordances already present on History and Day Detail surfaces, such as `同步這天紀錄中...`, `載入這天餐點中...`, and existing flat `SportCard` pending/empty/error containers.
- Same-day reconcile failure must not alter visible state; the correct visual result is no visible change.

## State Consistency Contract

| Surface | Required Behavior |
|---------|-------------------|
| Home / Summary today | Newer SSE totals appear only after same-day rows have refreshed successfully under the latest token. |
| Home meal rows | Rows must not be overwritten by stale initial loads after a newer SSE reconcile commits. |
| Summary detail today | Preserve existing loading/error/empty visuals; do not add new reconcile failure UI. |
| History selected week | Matching historical affected-date invalidation refreshes the selected day or visible week only. |
| History Day Detail | If open and `dateKey` matches `affectedDate`, refresh that detail view using existing loading/error/empty presentation. |
| Non-visible history dates | Passive invalidation only; no visible change until the user navigates to that date/surface. |

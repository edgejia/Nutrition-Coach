<objective>
Expose current meal revision identity on every edit-capable read and receipt surface.

Purpose: FRESH-01 requires edit-capable meal rows and chat receipts to carry read-side `mealRevisionId`, so clients can later send `expectedMealRevisionId` without inventing revision state.
Output: server read DTOs and chat receipt projections carry `mealRevisionId`; tests prove public `mealRevisionId` appears while internal `currentRevisionId` remains hidden.
</objective>

<decision_coverage>
- D-01: Add current `mealRevisionId` to `LoggedMealReceipt`, `/api/meals`-adjacent read rows, chat JSON/SSE `loggedMeal`, and restored history receipts when they can open Meal Edit.
- D-02: Preserve read/display `mealRevisionId` naming and avoid exposing write precondition naming in read DTOs.
- D-15: Restored current facts are the source for reopening Meal Edit; stale/non-current receipts must not remain edit-capable.
- D-16: Server-projected identity supports UX; it is not a substitute for write-side server precondition checks.
</decision_coverage>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SQLite -> API DTOs | Internal current revision pointers are projected into public read identities. |
| chat receipt storage -> browser | Persisted receipt-to-revision references are restored into edit-capable or display-only UI payloads. |
| server route -> SSE terminal payload | Chat receipt fields cross the streaming JSON boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-01 | Tampering | read DTOs | transfer | This plan exposes read identity only; write authority remains mitigated by Plan 01 and Plan 03 transaction precondition checks. |
| T-62-02 | Tampering | restored chat receipts | mitigate | Only current-active receipts receive `mealId`, `dateKey`, and `mealRevisionId`; stale/deleted receipts stay display-only. |
| T-62-03 | Elevation of privilege | `server/services/chat.ts` receipt lookup | mitigate | Keep joins constrained by `deviceId`, `chatMessages.deviceId`, `mealTransactions.deviceId`, and `mealRevisions.transactionId`. |
| T-62-04 | Information disclosure | public DTO projection | mitigate | Expose only public `mealRevisionId`; tests continue to deny `currentRevisionId`, raw revision history, prompts, user text, tool payloads, provider bodies, image data, session material, and DB snapshots. |
| T-62-05 | Tampering | client-only stale protection | transfer | This plan does not claim client read identity authorizes writes; Plan 01 and Plan 03 enforce server-side preconditions. |
| T-62-06 | Denial of service / Tampering | chat SSE projection | mitigate | Preserve existing SSE `status`/`chunk`/`done` order and only add a field to existing `loggedMeal` payloads. |
</threat_model>

<objective>
Make the direct meal mutation boundary revision-aware.

Purpose: FRESH-02 requires stale direct edit/delete attempts to fail before SQLite mutation, summary recompute, or realtime publish, while FRESH-01 requires direct edit-capable rows and successful update responses to carry the read-side revision identity.
Output: transaction-service precondition errors, direct `409` route bodies, direct DTO `mealRevisionId` projection, and focused unit/integration proof.
</objective>

<decision_coverage>
- D-01: Add `mealRevisionId` to `/api/meals` rows and direct update response meals.
- D-02: Use `expectedMealRevisionId` as the write precondition field, not `mealRevisionId`.
- D-03: Require `expectedMealRevisionId` for direct `PATCH` and direct `DELETE`.
- D-04: Leave meal creation/logging request bodies unchanged and do not accept `expectedMealRevisionId` there.
- D-05: Apply expected revision checks to stale deletes.
- D-06: Missing expected revisions fail closed with no mutation, no revision, no summary recompute, and no publish.
- D-07: Do not add a compatibility exception for missing expected revisions.
- D-08: Reject stale expected revisions before creating a new meal revision.
- D-09: Missing expected revision returns `409` with `error: "MEAL_REVISION_REQUIRED"`.
- D-10: Stale expected revision returns `409` with `error: "MEAL_REVISION_STALE"`.
- D-12: Reuse the route-owned `409` stable error string convention.
- D-16: Server-side checks are authoritative; client state is not trusted for write safety.
</decision_coverage>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> `server/routes/meals.ts` | Untrusted direct edit/delete JSON crosses into route validation and guest-session ownership checks. |
| route/service -> SQLite | Mutation request crosses into the authoritative meal revision write boundary. |
| route -> realtime publisher | Post-commit summary publish must not run after rejected writes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-01 | Tampering | `server/services/meal-transactions.ts` | mitigate | Compare `expectedMealRevisionId` against `meal_transactions.currentRevisionId` before any update/delete revision insert. |
| T-62-02 | Tampering | `server/routes/meals.ts` | mitigate | Missing `expectedMealRevisionId` returns `409 MEAL_REVISION_REQUIRED`; no compatibility fail-open branch is allowed. |
| T-62-03 | Elevation of privilege | `getActiveTransactionByDeviceAndId` | mitigate | Continue resolving by `{ deviceId, transactionId }` from signed guest sessions before comparing revision tokens. |
| T-62-04 | Information disclosure | direct conflict body/logs | mitigate | Return minimal metadata only: stable `error`, `mealId`, `affectedDate`, optional `currentMealRevisionId`; do not log raw prompt, user text, tool payloads, provider bodies, image data, session material, or DB snapshots. |
| T-62-05 | Tampering | client stale checks | mitigate | Treat client redaction as UX only; direct route writes rely on transaction-service precondition checks. |
| T-62-06 | Repudiation / Tampering | summary recompute and realtime publish | mitigate | Conflict catch branches return before `buildSummaryOutcomeAfterMealCommit` and `publishDailySummarySafe`; tests assert no side effects. |
</threat_model>

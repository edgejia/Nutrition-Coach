## Interaction Contract

### Receipt Edit Affordance

- A chat receipt is editable only when `buildReceiptMealEditPayload(...)` can produce `mealId`, `dateKey`, and `mealRevisionId`.
- If `mealRevisionId` is missing, do not show the chevron, do not set `role="button"`, and do not attach click/keyboard edit handlers.
- Do not add visible help text to old receipts. The receipt simply becomes display-only.
- Receipt keyboard behavior remains Enter/Space to open edit when editable.

### Meal Edit Save

- `MealEditPayload.mealRevisionId` is read-only UI state and must be sent as `expectedMealRevisionId` on save.
- While save is pending, keep current disabled inputs and `儲存中...` label.
- On `MEAL_REVISION_STALE` or `MEAL_REVISION_REQUIRED`, stop the stale save path, show the matching stale copy in the existing `role="alert"` error region, and trigger affected meal row refresh or invalidation.
- After stale refresh/invalidation starts, block additional save/delete attempts from that stale editor instance. The user must reopen Meal Edit from a refreshed row or receipt.

### Meal Edit Delete

- Delete requests for existing meals must include `expectedMealRevisionId` derived from the current `MealEditPayload.mealRevisionId`.
- Keep the existing browser confirmation copy.
- On stale conflict, do not close the screen as if delete succeeded. Show the stale-delete copy, refresh or invalidate the affected row/date, and disable stale delete/save actions until the user reopens from fresh state.

### Recovery Behavior

- Branch only on stable server error strings: `MEAL_REVISION_STALE` and `MEAL_REVISION_REQUIRED`.
- Client recovery is supportive only. The UI must never claim that client redaction or local freshness checks make a write safe.
- For current-day conflicts, refetch `/api/meals` with `refreshReason: "meal_mutation"` and update visible rows when available.
- For historical conflicts, record/invalidate the affected date and require the relevant day/detail surface to reload before another edit attempt.
- `redactChatReceiptIdentity(mealId)` may still be used after committed mutations and stale conflicts, but stale protection must not depend on redaction alone.

### Out Of Scope

- Do not design `/api/sse daily_summary` meal-row freshness; Phase 63 owns that.
- Do not add visible degraded-summary messaging from Phase 61 `summaryOutcome`.
- Do not add onboarding, marketing, help, or broad visual polish.
- Do not introduce shadcn, Radix, new registries, or a new icon library.

## Data And State Contract

| Surface | Required UI Contract |
|---------|----------------------|
| `LoggedMealReceipt` | Carries `mealRevisionId` whenever it can open Meal Edit. |
| `MealEntry` | Carries `mealRevisionId` for home/history/day-detail rows that can open Meal Edit. |
| `MealEditPayload` | Carries `mealRevisionId`; never exposes `expectedMealRevisionId` as display data. |
| `UpdateMealInput` | Sends `expectedMealRevisionId` derived from the payload. |
| `DeleteMeal` client call | Sends `expectedMealRevisionId` derived from the payload. |
| `api.ts` errors | Preserve `error`, `mealId`, `affectedDate`, and optional `currentMealRevisionId` for stale conflict handling. |
| `store.ts` | Owns refresh/invalidation and stale receipt redaction; does not decide mutation authority. |

<objective>
Thread expected meal revision identity through chat/tool update and delete flows.

Purpose: FRESH-02 covers stale writes from older chat receipts and model/tool correction flows, so chat `update_meal` and `delete_meal` must not mutate by meal id alone after a target has been resolved.
Output: resolver-owned revision targets, tool-session state carrying `{ mealId, mealRevisionId }`, service/tool pass-through to the transaction precondition boundary, and tests proving stale tool writes fail closed.
</objective>

<decision_coverage>
- D-02: Preserve `expectedMealRevisionId` as the write precondition name.
- D-03: Require `expectedMealRevisionId` for chat/tool `update_meal` and `delete_meal` after target meal resolution.
- D-05: Apply the contract to stale deletes as well as stale edits.
- D-06: Missing/stale preconditions fail closed with no mutation, no revision, no summary recompute, and no publish.
- D-07: Do not add legacy compatibility for unresolved or missing expected revisions.
- D-08: Reject stale expected revisions before the transaction write boundary creates a new revision.
- D-16: Server-side resolver and transaction checks are authoritative; model/client state is not trusted.
</decision_coverage>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM tool args -> orchestrator tools | Model-authored `meal_id` and tool args cross into backend tool execution. |
| orchestrator tools -> meal correction service | Resolved target identity is transformed into a mutation precondition. |
| meal correction service -> transaction service | Chat/tool update/delete crosses into authoritative SQLite revision writes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-01 | Tampering | `update_meal` / `delete_meal` tools | mitigate | Require resolver-owned `{ mealId, mealRevisionId }` and pass `expectedMealRevisionId` to transaction writes. |
| T-62-02 | Tampering | missing expected revision in tool path | mitigate | Remove id-only `resolvedMealIds` compatibility and fail closed when no resolved revision target exists. |
| T-62-03 | Elevation of privilege | `mealCorrectionService` target lookup | mitigate | Continue resolving candidates by device-owned active meals and do not compare revision ids without matching `deviceId` and `mealId`. |
| T-62-04 | Information disclosure | tool failure logs/proof | mitigate | Preserve stable error codes only; do not log raw prompts, user text, assistant final text, tool raw payloads, provider bodies, image data, session material, or DB snapshots. |
| T-62-05 | Tampering | client-only stale checks | mitigate | Tool path does not rely on client receipt redaction; backend resolver and transaction preconditions decide authority. |
| T-62-06 | Repudiation / Tampering | mutation receipt and summary/publish side effects | mitigate | Stale tool failures do not emit committed mutation receipts, `summaryOutcome`, recompute calls, or realtime publishes. |
</threat_model>

<objective>
Make the client carry revision identity into edit/delete requests and recover deterministically from stale conflicts.

Purpose: FRESH-03 requires user-visible stale guidance and row refresh/invalidation after server-side stale rejection, while FRESH-01 requires client edit payloads and receipts to carry read-side revision identity.
Output: client DTOs, API helpers, edit payload builders, receipt affordance gating, stale-conflict UI behavior, and focused client tests.
</objective>

<decision_coverage>
- D-01: Add `mealRevisionId` to `LoggedMealReceipt`, `MealEntry`, and `MealEditPayload`.
- D-02: Send `expectedMealRevisionId`, not `mealRevisionId`, in write inputs.
- D-03: Require `expectedMealRevisionId` for client direct update/delete calls.
- D-11: Branch on stable `MEAL_REVISION_REQUIRED` and `MEAL_REVISION_STALE` and show deterministic Traditional Chinese copy.
- D-13: On stale conflict, show guidance, close/block saving from stale editor/receipt, and refresh or invalidate affected rows.
- D-14: Perform direct client reaction to `409`; do not implement Phase 63 SSE meal-row freshness.
- D-15: Require reopening Meal Edit from refreshed current facts after a stale conflict.
- D-16: Treat client behavior as UX support only; server checks remain authoritative.
</decision_coverage>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server `409` response -> client API | Untrusted response parsing must preserve stable conflict codes without exposing sensitive payloads. |
| client edit payload -> direct routes | Client sends stale-prone revision preconditions to server authority. |
| store/UI -> user action | UI blocks stale editor reuse but must not be mistaken for write authority. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-62-01 | Tampering | `client/src/api.ts` write calls | mitigate | Serialize `expectedMealRevisionId` from `MealEditPayload.mealRevisionId`; server still enforces authority. |
| T-62-02 | Tampering | missing expected revision from client | mitigate | Payload builders require `mealRevisionId`; write helpers require `expectedMealRevisionId`; server `MEAL_REVISION_REQUIRED` remains authoritative. |
| T-62-03 | Elevation of privilege | forged/foreign device mutation attempts | transfer | Client cannot authorize ownership; Plans 01 and 03 preserve signed cookie/device-scoped server lookups. |
| T-62-04 | Information disclosure | conflict errors and tests | mitigate | Preserve only stable error code and minimal meal metadata; tests must not store raw prompt, user text, assistant final text, tool payloads, provider bodies, image data, session material, or DB snapshots. |
| T-62-05 | Tampering | client-only stale protection | mitigate | UI copy and blocked controls are labeled as recovery behavior; source relies on server `409` conflicts for authority. |
| T-62-06 | Repudiation / Tampering | stale conflict recovery | mitigate | Stale conflicts trigger row refresh/invalidation and do not call success navigation paths or mutation success copy. |
</threat_model>

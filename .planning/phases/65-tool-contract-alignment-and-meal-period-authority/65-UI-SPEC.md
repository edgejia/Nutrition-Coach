## Meal-Period Presentation Contract

Public DTO field: `mealPeriod?: "breakfast" | "lunch" | "dinner" | "late_night"`.

Label mapping:

| Enum | Label | Home badge |
|------|-------|------------|
| `breakfast` | `早餐` | `B` |
| `lunch` | `午餐` | `L` |
| `dinner` | `晚餐` | `D` |
| `late_night` | `宵夜` | `N` |
| missing / invalid | fallback from `loggedAt`; otherwise `餐點` | fallback badge; otherwise `M` |

Fallback labels from `loggedAt` must preserve current legacy behavior: `05:00-10:59` maps to `早餐`, `11:00-13:59` maps to `午餐`, `14:00-16:59` maps to `點心`, `17:00-22:59` maps to `晚餐`, and all invalid/missing/outside-hour values map to `餐點`.

Interaction rule: explicit `mealPeriod` always wins over hour inference. A meal logged at breakfast hour with `mealPeriod: "lunch"` must display `午餐` everywhere that Phase 65 touches.

## Surface Contract

| Surface | Required visual behavior |
|---------|--------------------------|
| `HomeScreen` meal row | Keep existing row layout. Render `HH:mm`, resolved meal-period label, and badge using `mealPeriod` first and `loggedAt` fallback second. |
| `HistoryScreen` meal row | Keep existing timeline row layout. Change metadata from time-only to `HH:mm · {meal label}` using the same resolver. |
| `HistoryDayDetailScreen` meal row | Keep existing detail card layout. Change time metadata from time-only to `HH:mm · {meal label}` using the same resolver. |
| `SummaryDetailScreen` meal row | Keep existing summary row layout. Change time metadata from time-only to `HH:mm · {meal label}` using the same resolver. |
| Chat logged-meal receipts and edit entry points | Carry `mealPeriod` in DTO/payload state when present, but do not add new visible controls. |

Accessibility: visible meal labels and accessible row names must agree. When a meal row has an edit button, its `aria-label` should include the resolved meal label before the food name, for example `編輯 午餐 雞腿便當`.

## DTO And Interaction Contract

- Add `mealPeriod` to `LoggedMealReceipt`, `MealEntry`, `MealEditPayload`, and client normalizers.
- `client/src/api.ts` must accept only the four enum values; invalid values are treated as absent.
- Do not add `inferredMealPeriod` to public client types.
- Do not send an inferred fallback as `mealPeriod`.
- `client/src/meal-edit-payload.ts` must preserve explicit `mealPeriod` from a meal row or logged receipt when building edit state.
- Ordinary edit submits must not clear an existing backend `mealPeriod` when the edit omits period changes.
- Do not add a meal-period picker, correction modal, confirmation dialog, snackbar, or toast in Phase 65.
- No loading, empty, or error state changes are required beyond preserving existing behavior after DTO normalization.
